O que é isso?

Este guia pretende oferecer sugestões e recomendações para a construção de elementos de interface gráfica em sistemas web.

Acreditamos que fornecendo um repositório de informações centralizado, as equipes de desenvolvimento poderão recorrer a um ponto comum, obtendo assim uma padronização natural do comportamento e das funcionalidades de cada componente.

Página Inicial - Landing Page

Para o caso de ferramentas e utilitários para o suporte às atividades de seus usuários, precisamos de uma página inicial objetiva.

Sugerimos a inclusão de um pequeno texto contendo um overview das principais funcionalidades do sistema. Este recurso auxiliará usuários que não estão certos do sistema que procuram. Este texto também pode conter instruções para os primeiros passos do usuário no sistema, como a ação de login por exemplo

Em caso de sistemas que requeiram autenticação, é importante que as funcionalidades do sistema não sejam expostas, mesmo que estejam bloqueadas para o acesso do usuário não autorizado. Isto porque, a experiência de acesso negado gera frustração aos usuários. Assim, se o usuário em questão não tem acesso, melhor omití-lo que proibí-lo.

Podemos também incluir o formulário de login na página inicial.

Barra de Navegação Global

A navegação global tem mais relação com a taxonomia das informações a serem exibidas que com suas características técnicas. Uma boa navegação é constituída por itens equivalentes em cada nível. Por exemplo: Se temos um item "Mamífero" no primeiro nível, um segundo item deveria ser "Felino" e não "Gato" pois este último deveria estar no nível imediatamente inferior juntamente com os tigres e leões.

Feita esta observação, em termos de componentes, devemos usar items genéricos no primeiro nível e como em webapps o conteúdo não é muito extenso, então ómitir os niveis inferiores é bem razoável (em portais de conteúdo, pode-se utilizar níveis inferiores com muitos itens e recursos visuais ou exibir níveis inferiores logo de início).

Não há uma regra definida no que se refere à quantidade de níveis de menu devemos manter. Contudo, é notório que menus muito sobrecarregados são fontes de confusão para os usuários. Caso a hierarquia seja estensa, é melhor utilizar-se de outros componentes de navegação para complementar a barra de navegação global como por exemplo o Menu Lateral.

Itens que podem estar presentes na barra de navegação global:

  • Logomarca (Brand);
  • Itens de menu globais podendo ter, ou não, sub-itens;
  • Campo de pesquisa (com função full-text search);
  • Notificações;
  • Menu de configuração de conta, com, ou sem dropdown;
  • Label de versão.

Formulário de Login

Formulários de login são componentes muito comuns e a grande maioria dos usuários que possuem alguma experiência com sistemas web podem reconhecê-los e operá-los fácilmente. No entanto, definir um padrão para estes componentes pode reduzir o gap entre sistemas diferentes.

Itens comuns em formulários de login são:

  • Labels;
  • Text fields;
  • Botão com a ação do formulário;
  • Link para recuperação de senha;
  • Link para cadastro de conta;
  • Checkbox para a persistência da sessão.

Contudo, os itens citados variam de acordo com a natureza do sistema. Em sistemas com cadastro de usuários restrito, não haverá link para cadastro de conta; sistemas com altos requisitos de segurança não devem fornecer persistência de sessão; etc.

Segundo artigo publicado no site Ux Matters, formulários devem trazer seus rótulos acima dos campos de entrada pois usuários fazem a leitura no sentido top-down e têm mais facilidade em associar o rótulo com o campo abaixo do que com o campo ao lado.

É importante que os rótulos e o padrão visual dos campos e botões sejam consistentes entre os sistemas da organização. Devemos utilizar o rótulo "Entrar" para todos os botões de login dos sistemas. Também devemos destacar oo botão com a cor primária de cada sistema.

Campo de Pesquisa

Definimos Campo de Pesquisa (Search Box = input field + submit button) como aquele componente utilizado para realizar buscas no contexto global da aplicação. Geralmente, este campo é utilizado para mecanismos de busca full-text.

Em alguns casos, quando a aplicação trata de uma entidade de domínio principal específica, o Campo de Pesquisa poderá estar em um escopo mais restrito, mas, via de regra, campos de pesquisa possuem escopo global.

Componentes de pesquisa com escopos específicos são componentes de filtragem. Exemplos de search box atuando como filtros geralmente são encontrados em tabelas e grids de dados.

Campos de pesquisa devem estar claramente visíveis, reconhecíveis e fáceis de usar. Para alcançar estas três caracteríscas sugerimos a utilização de um botão com ícone lupa acoplado ao input field, a utilização de um placeholder com o texto "Busca" e utilização de recurso live search.

Estudo realizado por Jacob Nielsen mostrou que o tamanho médio de campos de pesquisa existentes era de 18 caracteres e que, com este tamanho, 27% das consultas eram maiores que o tamanho máximo do campo. O estudo mostrou também que utilizando campos de pesquisa com o tamanho de 27 caracteres, 90% das consultas estarão adequadas ao tamanho do campo.

Pontos relevantes:

  • A largura não deve ser muito grande para não atrair toda a atenção ao componente, e nem deve ser muito pequena pois assim induz o usuário a utilizar termos mais curtos e imprecisos.
  • O posicionamento deve ser fixo e se aproveitar dos padrões mais comumente utilizados.
  • Botões muito pequenos desestimulam a utilização da pesquisa pois obriga um posicionamento muito preciso do cursor.
  • Evitar colocá-lo próximo a outros formulários e inputs.
  • Preze pela simplicidade. Evite sobrecarregar visualmente o componente.
  • Em caso de label de texto, utilize-o consistentemente. Sugestão: "Pesquisar"
  • Utilizar design exclusivo, ou seja, nenhum outro componente com funcionalidade diferente deverá se parecer com um Campo de Pesquisa.
  • Utilize apenas 1 botão.

Filtros em Listagens

Filtros são componentes extremamente úteis em listagens de dados, principalmente quando lidamos com grandes volumes. Sua função é bastante intuitiva: filtrar os dados exibidos de acordo com os caracteres inseridos.

As principais características para estes componentes são:

  • Distinção visual do componente (geralmente com associação a um ícone lupa);
  • Filtragem automática e em "tempo real". É desejável que a filtragem seja disparada pelo evento onkeyup para que o usuário tenha feedback imediato.
  • Em alguns casos, quando o esforço de implentação seja pequeno, torna-se interessante atribuir um highlight nos termos da listagem que "casam" com o termo inserido.
  • Também é desejável que se permita a filtragem considerando qualquer campo de texto exibido no grid.
  • É importante que os filtros sejam visualmente distintos dos campos de pesquisa.

Campo de Seleção com Sugestão

Conhecidos como select, combobox, dropdowns, os campos de seleção são muito úteis em formulários. Em muitos dos casos, exibem itens estáticos, finitos e razoavelmente localizáveis. Porém, há casos em que podemos ter muitos itens de modo que se torna inviável a localização do item apenas rolando a listagem. E ainda, outros casos em que o tamanho da listagem é desconhecido pois esta apresenta itens dinâmicos que podem aumentar muito com o tempo.

Para estes últimos casos, foram criadas alternativas para facilitar o seu uso. Conhecidos como autocomplete field, typeahead, suggestion boxes, daremos a este componente o nome de Select com Sugestão pois associa o componente select com um recurso que sugere e filtra a listagem à medida em que o usuário digita no campo.

Visualmente, algumas características são desejáveis neste componente:

  • Visualização em contexto: o usuário deve filtrar sem sair da página ou ter seu foco desviado.;
  • Distinção visual clara: o usuário deve ser capaz de reconhecer que aquele campo de seleção tem recurso adicional;
  • Tempo de resposta razoávelmente pequeno;
  • Feedback visual como o uso de "spinners" por exemplo;
  • Coerência na filtragem: os itens sugeridos devem estar de acordo com os caracteres digitados.

É importante que todos os campos de todos os sistemas que tenham demandas semelhantes às descritas acima, apresente funcionalidade e modo de uso consistentes.

Existem diversas bibliotecas open source em javascript que podem ser adicionados ao front-end dos aplicativos. Para ilustrar um componente de Seleção com Sugestão, utilizamos o Select2 com o estilo fornecido por select2-bootstrap-css acessando a API pública de RottenTomatoes.com.

Grid de Dados

Grids de dados são elementos de interface gráfica utilizados para dispor listas de informações. Geralmente são dispostos em formato de tabela contendo linhas e colunas.

Os grids são muito comuns em seções de cadastro (CRUD's) para listar os objetos existentes de uma determinada entidade.

Os recursos de usabilidade desejáveis em um grid de dados são:

  • Simplicidade no design: por se tratar de um item naturalmente carregado de informações, suavizar o design mantendo o menor número de distrações possível na interface, ameniza a experiência do usuário.
  • Ordenação: a ordenação é útil quando o usuário não tem um critério de busca preciso, por exemplo.
  • Filtragem: listas dinâmicas rapidamente demandam maneiras de refinar seus conteúdos para facilitar a busca de itens.
  • Ações: com o formato de tabela, ações globais e individuais tornam-se bastante intuitivas. As ações globais devem ser adicionadas à tabela em um cabeçalho, por exemplo. E as ações individuais devem ser adicionadas a uma célula de cada item, geralmente na última coluna.
  • Seleção: quando há a necessidade de ações em massa como remover, mover, alterar vários itens (mas não todos) de uma só vez, é necessário permitir a seleção dos itens a aplicar as ações. A seleção deve ser visualmente identificável e, via de regra, deve ser inserida na primeira coluna.
  • Paginação: Listas muito extensas tornam-se incômodas visualmente. Assim, devemos incluir um recurso para paginar os resultados. O ideal é que a paginação esteja visível sem a necessidade de rolar a página. Se a listagem for muito extensa, pode-se utilizar dois elementos de paginação: no cabeçalho e no rodapé da tabela.

O exemplo abaixo foi construído apenas no cliente utilizando as bibliotecas Underscore.js, Backbone.js e Backgrid.js com a finalidade de exemplificar como um componente Grid de Dados deve se parecer.

        name: "percentage",

        cell: "number"

Gerenciamento de Conta

Os sistemas que demandam autenticação geralmente oferecem recursos para o gerenciamento de contas de usuário. Este gerenciamento pode incluir inúmeras funcionalidades como troca de senha, troca de e-mail, upload de fotos e várias outras relativas à administração.

Nos últimos anos, tem se tornado um padrão entre as diversas aplicações web o uso de um elemento dropdown no canto superior direito da aplicação. Muitas vezes, as opções de gerenciamento ficam associadas ao nome de usuário pois esta disposição torna intuitiva a associação da conta do usuário com as tarefas administrativas desta.

Aproveitar um padrão emergente é desejável pois reduz o impacto na transição do usuário. Assim, sugerimos as seguintes características:

  • Exibição do nome de usuário com indicativo de dropdown;
  • Se possível, associar a foto do usuário ou ícone genérico
  • Usar dropdown com separadores quando houver mais de uma categoria de configurações. (Conta pessoal, segurança, parametrização, ajuda, etc.)
  • Utilizar a função de logout aninhada no dropdown

Upload de Arquivo

São elementos de formulário que permitem a seleção de arquivos locais para envio ao servidor. Os navegadores possuem campos de upload nativos e os renderizam sempre que encontrarm um elemento "input" do tipo "file". Em termos de usabilidade, a principal vantagem de se utilizar o elemento nativo é se aproveitar da experiência já adquirida pelo usuário em outros sistemas na web. Contudo, o campo de upload nativo não oferece muitas funcionalidades como barra de progresso, upload em mais de uma etapa, remoçao do arquivo adicionado, entre outras.

Desta forma, sempre que houver necessidade de alguma funcionalidade adicional, será necessário estender o comportamento nativo ou integrar alguma biblioteca existente como a jQuery File Upload por exemplo.

A principal característica desejável em um campo de upload de arquivo é sua distinção visual e intuitividade. Para isso, é interessante que utilizemos um padrão para o rótulo e também a utilização de um ícone sugestivo também padronizado para os botões:

O uso de barra de progresso também melhora a usabilidade no quesito feedback, pois informa o usuário sobre a situação do upload em questão.

O upload em duas etapas aumenta reduz a propensão a erros reduzindo uploads indesejados. Também permite a exclusão de determinado antes de enviá-lo ao servidor reduzindo desperdícios onde o gargalo é maior.

Como um campo de upload completo demanda a utilização de alguma tecnologia no servidor (ausente neste manual), esta seção não contará com um exemplo, porém, este pode ser encontrado na página da biblioteca citada nos parágrafos anteriores.

Ajuda de Contexto

Este elemento é muito útil em formulários, principalmente formulários extensos, com campos específicos de negócio.

É importante notar que, este componente só será útil se for apresentado, de fato, no contexto em que é necessário. Ou seja, não devemos redirecionar o usuário para uma página de ajuda ou algo do tipo. O ideal é apresentar a ajuda em um popover, tooltip ou até mesmo em um modal.

Quanto ao conteúdo da ajuda, o ideal é ser sucinto e objetivo. Restringir a informação apresentada às dúvidas que, por ventura, possam ocorrer durante o preenchimento do campo.

Para o componente, sugerimos que seja disponibilizado (quando houver ajuda disponível) um ícone sugestivo ao lado do rótulo do campo.

Exemplo

Código do exemplo:

    <form>
      <div class="form-group">
        <label for="ex">Rótulo do Campo 
          <a id="popover" role="button"><i class="fa fa-question-circle"></i></a>
        </label>
        <input type="text" class="form-control" name="ex" id="ex">    
      </div>      
    </form>
    <script>
      $('document').ready(function() {
        $('#popover').popover({
          html: true,
          content: '<p style="font-weight: 100">O <em>popover</em> suporta até mesmo conteúdo <b>HTML</b></p>',
          title: 'Título da <strong>Ajuda</strong>'
        });  
      });      
    </script>

Componente "Loading"

Em processamentos longos, onde o carregamento de uma página ou recurso leva um tempo acima da média, é necessário fornecer algum recurso que dê feedback ao usuário indicando que a tarefa está sendo processada.

Como características desejáveis sugerimos:

  • que utilizemos o mesmo rótulo em todos os elementos deste tipo e em todos os sistemas;
  • utilizar algum componente animado, pois assim fica ainda mais claro que alguma atividade está sendo realizada;
  • bloquar a tela em segundo plano para impedir a interação do usuário enquanto o processamento é realizado;
  • utilizar janela modal em tamanho fixo e não permitir redimensionamento.
